# 游标分页

## 什么是游标分页

游标分页(Cursor Pagination)是一种基于游标指针的分页方式，通过唯一标识符来定位数据位置，而不是使用偏移量(offset)。游标通常是记录的某个唯一字段(如ID、时间戳)的编码值。

## 为什么需要游标分页

### 传统偏移分页的问题

```sql
-- 偏移分页
SELECT * FROM posts ORDER BY created_at DESC LIMIT 10 OFFSET 20;
```

**主要问题:**

1. **数据一致性问题**: 翻页过程中如果有新数据插入或删除，会导致数据重复或遗漏
2. **性能问题**: offset越大，数据库需要扫描跳过的行数越多，性能越差
3. **深分页困难**: 当offset达到百万级别时，查询会非常慢

### 游标分页的优势

1. **稳定性**: 不受数据变化影响，每次查询基于上次的位置
2. **性能稳定**: 无论在第几页，查询性能都是O(1)级别
3. **适合实时数据流**: 天然支持"加载更多"的场景

## 实现原理

### 基本思路

使用上一次查询结果的最后一条记录的唯一标识作为"游标"，下次查询从这个位置继续。

### SQL实现

```sql
-- 第一页
SELECT * FROM posts
ORDER BY created_at DESC, id DESC
LIMIT 11;

-- 后续页(假设上一页最后一条: created_at='2024-01-01', id=100)
SELECT * FROM posts
WHERE (created_at, id) < ('2024-01-01', 100)
ORDER BY created_at DESC, id DESC
LIMIT 11;
```

查询 limit+1 条，用于判断是否还有下一页。

## API设计

### 请求格式

```typescript
interface PaginationRequest {
  cursor?: string; // base64编码的游标
  limit: number;
}
```

### 响应格式

```typescript
interface PaginationResponse<T> {
  data: T[];
  pageInfo: {
    hasNextPage: boolean;
    hasPreviousPage: boolean;
    endCursor: string;
  };
}
```

### 完整实现

```typescript
async function getPosts(cursor?: string, limit: number = 10) {
  const decodedCursor = cursor
    ? JSON.parse(Buffer.from(cursor, "base64").toString())
    : null;

  const whereClause = decodedCursor
    ? sql`WHERE (created_at, id) < (${decodedCursor.created_at}, ${decodedCursor.id})`
    : sql``;

  const posts = await db.query(sql`
    SELECT * FROM posts 
    ${whereClause}
    ORDER BY created_at DESC, id DESC 
    LIMIT ${limit + 1}
  `);

  const hasNextPage = posts.length > limit;
  const data = hasNextPage ? posts.slice(0, limit) : posts;
  const lastItem = data[data.length - 1];

  const endCursor = lastItem
    ? Buffer.from(
        JSON.stringify({
          created_at: lastItem.created_at,
          id: lastItem.id,
        }),
      ).toString("base64")
    : null;

  return {
    data,
    pageInfo: { hasNextPage, hasPreviousPage: !!cursor, endCursor },
  };
}
```

## 关键技术点

### 1. 复合游标

排序字段可能有重复值，必须添加唯一字段保证顺序确定:

```sql
-- 错误: created_at可能重复，导致丢数据
ORDER BY created_at DESC

-- 正确: 添加id保证唯一性
ORDER BY created_at DESC, id DESC
```

### 2. 游标编码

避免泄露内部结构，对游标进行base64编码:

```typescript
const encodeCursor = (obj: any) =>
  Buffer.from(JSON.stringify(obj)).toString("base64");

const decodeCursor = (cursor: string) =>
  JSON.parse(Buffer.from(cursor, "base64").toString());
```

### 3. 索引优化

```sql
-- 复合索引，顺序要和ORDER BY一致
CREATE INDEX idx_posts_cursor ON posts(created_at DESC, id DESC);
```

### 4. 双向分页

```sql
-- 向后翻页
WHERE (created_at, id) < (cursor_time, cursor_id)
ORDER BY created_at DESC, id DESC

-- 向前翻页(反转条件和排序)
WHERE (created_at, id) > (cursor_time, cursor_id)
ORDER BY created_at ASC, id ASC
-- 注意: 结果需要reverse回来
```

## 适用场景

### 适合游标分页

- 社交媒体时间线(Twitter、Instagram)
- 无限滚动加载
- 实时聊天消息列表
- 数据导出/ETL批量任务
- 高并发、大数据量场景

### 不适合游标分页

- 需要跳转到任意页码(如管理后台)
- 需要显示"第X页 / 共Y页"
- 数据量小且变化不频繁

## GraphQL Relay规范

GraphQL社区定义了标准的游标分页格式:

```graphql
type Query {
  posts(first: Int, after: String, last: Int, before: String): PostConnection!
}

type PostConnection {
  edges: [PostEdge!]!
  pageInfo: PageInfo!
}

type PostEdge {
  node: Post!
  cursor: String!
}

type PageInfo {
  hasNextPage: Boolean!
  hasPreviousPage: Boolean!
  startCursor: String
  endCursor: String
}
```

## 常见陷阱

### 1. 游标字段非唯一

如果只用 `created_at` 做游标，两条记录时间相同时会丢数据。务必用复合字段。

### 2. 游标失效

记录被删除后，游标指向的位置可能不存在。使用 `<` / `>` 而非 `=` 来查询，天然兼容这种情况。

### 3. 排序方向与比较符号匹配

- DESC排序 → 使用 `<`
- ASC排序 → 使用 `>`

搞反了会查不到数据或返回错误方向的结果。

## 性能对比

| 分页方式 | 第1页 | 第100页 | 第10000页 |
| -------- | ----- | ------- | --------- |
| Offset   | ~1ms  | ~10ms   | ~500ms    |
| Cursor   | ~1ms  | ~1ms    | ~1ms      |

offset分页的性能随页码线性退化，游标分页始终稳定。

## 总结

**选择建议:**

- 小数据量 + 需要页码跳转 → offset分页
- 大数据量 + 实时流 + 只需前后翻页 → 游标分页
- 两者都需要 → 提供两套API，不同场景使用不同方案
